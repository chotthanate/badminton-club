# AGENTS.md

กติกานี้ใช้กับทั้งโปรเจกต์ เพื่อให้สลับทำงานระหว่าง PC และ Mac ผ่าน GitHub ได้อย่างปลอดภัย โดยสมมติว่าจะไม่ทำงานสองเครื่องพร้อมกัน

## เมื่อผู้ใช้บอกว่า “เก็บงาน”

ให้ทำตามลำดับนี้:

1. ตรวจสอบสถานะด้วย `git status --short --branch` และอ่าน diff ทั้งไฟล์ที่แก้ ไฟล์ใหม่ และไฟล์ที่ stage แล้ว เพื่อยืนยันขอบเขตงาน
2. ตรวจหา passwords, API keys, tokens, `.env`, credentials, private keys, certificates และไฟล์ลับก่อน stage และก่อน commit
   - ห้าม commit ไฟล์ `.env*` ที่มีค่าจริง ยกเว้นไฟล์ตัวอย่างที่มีแต่ placeholder เช่น `.env.example`
   - ถ้าพบค่าที่อาจเป็นความลับ ให้หยุดและอธิบายผู้ใช้ ห้าม push จนกว่าจะจัดการเรียบร้อย
3. stage เฉพาะไฟล์ที่เป็นงานรอบนี้ ห้ามใช้ `git add -A` เมื่อมีไฟล์ที่ไม่เกี่ยวข้องหรือยังไม่ทราบเจ้าของ
4. รันทดสอบที่เหมาะกับสิ่งที่เปลี่ยน อย่างน้อยใช้ `npm test` และ `npm run build` เมื่อแก้โค้ดเว็บ หากรันไม่ได้ให้แจ้งเหตุผลชัดเจน
5. สร้างหรืออัปเดต `WORK_STATUS.md` เฉพาะเมื่อยังมีงานค้าง หรือมีบริบทสำคัญที่เครื่องถัดไปดูจากโค้ดและ commit ไม่ออก โดยระบุสิ่งที่เสร็จแล้ว งานค้าง วิธีทดสอบ และข้อควรระวังแบบสั้น ๆ
6. ใช้ `git fetch origin` ตรวจสอบ remote ก่อนบันทึก หาก branch ปัจจุบันตามหลังหรือแยกจาก remote ให้หยุดและอธิบายผู้ใช้ ห้าม merge หรือ rebase เพื่อแก้เอง
7. commit ด้วยข้อความสั้นที่อ่านแล้วเข้าใจว่าแก้อะไร จากนั้น push แบบปกติไปยัง GitHub
8. แจ้งผู้ใช้ว่า commit และ push สำเร็จหรือไม่ พร้อมผลทดสอบและสิ่งที่ยังค้าง

## เมื่อผู้ใช้บอกว่า “ทำงานต่อ”

ให้ทำตามลำดับนี้ก่อนแก้ไฟล์ใด ๆ:

1. ตรวจด้วย `git status --short --branch` รวมทั้งไฟล์ untracked
2. หากมีไฟล์แก้ไข ไฟล์ stage หรือไฟล์ใหม่ที่ยังไม่ได้บันทึก ให้หยุดและอธิบายผู้ใช้ก่อน ห้ามลบ เขียนทับ stash หรือแก้ไฟล์เหล่านั้นเอง
3. เมื่อ working tree สะอาด ให้ใช้ `git fetch origin` แล้วดึงเฉพาะแบบ fast-forward ด้วย `git pull --ff-only`
4. หาก pull ไม่สำเร็จ มี conflict หรือประวัติ branch แยกกัน ให้หยุดและอธิบายผู้ใช้แบบเข้าใจง่าย
5. อ่าน commit ล่าสุดด้วย `git log -1 --stat` และอ่าน `WORK_STATUS.md` หากมี
6. หาก `package-lock.json` เปลี่ยนจาก commit ที่เครื่องนี้เคยมี ให้รัน `npm ci` ก่อนทดสอบหรือพัฒนา
7. สรุปสั้น ๆ ให้ผู้ใช้ว่างานล่าสุดทำอะไร เสร็จถึงไหน และมีงานค้างอะไร แล้วจึงทำงานต่อจากจุดนั้น

## กฎความปลอดภัยและการแก้ปัญหา

- ห้ามใช้ force push ทุกรูปแบบ รวมถึง `--force-with-lease`
- ห้ามใช้คำสั่งทำลายงาน เช่น `git reset --hard`, `git clean -fd` หรือ checkout/restore เพื่อทิ้งงานของผู้ใช้ เว้นแต่ผู้ใช้สั่งอย่างชัดเจนหลังได้รับคำอธิบายผลกระทบ
- ห้ามแก้ conflict, rebase, merge, ลบ branch หรือเขียนทับไฟล์เพื่อให้ผ่านเองเมื่อสถานการณ์ไม่แน่ใจ ให้หยุดและถามผู้ใช้ก่อน
- รักษา `.gitignore` ให้กันไฟล์ลับ ไฟล์ build, dependencies, logs, coverage, ไฟล์ระบบปฏิบัติการ และไฟล์ตั้งค่า editor เฉพาะเครื่อง
- ค่าที่หน้าเว็บจำเป็นต้องเปิดเผยต่อ browser ให้เก็บเป็น GitHub Actions repository variables สำหรับการ deploy ไม่เก็บใน `.env.production` บน GitHub ส่วน secrets ฝั่ง server ให้เก็บใน Supabase/GitHub Secrets เท่านั้น
- เมื่อเครื่องใหม่ต้องรันเว็บแบบ local และยังไม่มี `.env.local` ให้ดึงเฉพาะ repository variables ที่จำเป็นผ่าน GitHub CLI ที่ล็อกอินแล้ว หรือขอค่าจากผู้ใช้ จากนั้นสร้าง `.env.local` เฉพาะในเครื่องโดยไม่แสดงค่าลง log และตรวจว่า Git ignore ไฟล์นี้จริงก่อนทำงาน
- ก่อน push ให้ตรวจทั้งรายชื่อไฟล์ที่กำลัง commit และเนื้อหา diff อีกครั้ง
- อย่าใส่ secret จริงลง `WORK_STATUS.md`, commit message, PR description, log หรือข้อความตอบผู้ใช้
